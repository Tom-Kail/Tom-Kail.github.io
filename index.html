golang channel 源码学习

发表于 2018-05-20 |

概述

channel是Go语言在语言级别提供的goroutine间的通信方式。我们可以使用channel在两个或多个goroutine之间传递消息。channel是进程内的通信方式，因此通过channel传递对象的过程和调用函数时的参数传递行为比较一致，比如也可以传递指针等。

使用方法

源码

主要结构体

等待队列，是个链表

主要方法

makechan

初始化channel的方法

chansend

往channel里发送数据，如果有等待中的接收者，则将数据直接发送给接收者，否则，如果有可用缓存空槽，则将数据放到空槽中，若没有等待者也没有空槽，则阻塞在那里

chanrecv

从channel里接收数据，如果buffer里面有数据，先从buffer队列的头部取出数据，如果buffer为空，且有等待发送数据的发送者，直接将该发送者的数据拷贝到接收者的内存中，若果buffer为空且没有等待发送数据的发送者，接收者阻塞着

IO多路复用与go

发表于 2018-05-05 |

概述

I/O 多路复用技术是为了解决进程或线程阻塞到某个 I/O 系统调用而出现的技术，使进程不阻塞于某个特定的 I/O 系统调用。
select、poll和epoll都是I/O多路复用的机制。I/O多路复用通过一种机制，可以监视多个描述符，一旦某个描述符就绪（一般是读就绪或者写就绪，就是这个文件描述符进行读写操作之前）
与多线程和多进程相比，I/O 多路复用的最大优势是系统开销小，系统不需要建立新的进程或者线程，也不必维护这些线程和进程。

sync.Map源码分析

发表于 2018-04-25 |

概述

go语言中的map并不是并发安全的,在Go 1.6之前,并发读写map会导致读取到脏数据,在1.6之后则程序直接panic.

go官方博客中给出一种方案。

有网友写了concurrent-map

这个包支持并发安全,采用sharding的方式。他向go官方推荐了该方案，但被拒了。

原理

为了减少并发抢锁导致的阻塞，sync.Map分出了read和dirty两个map，里面存的都是指针。存、删和查都先操作read，并用atomic进行并发保护，速度较快，直到read不能满足需求才去操作dirty，操作dirty的时用Mutex锁进行并发保护，速度较慢。

源码分析

主要结构

entry

用于保存value的interface指针,通过atomic进行原子操作

readOnly

Map.read 用的就是readOnly，对其进行操作的时候，使用atomic进行保护

Map

主要方法

取数据

存数据

Range方法

通过回调函数f访问map中的数据，Range内部循环将key，value传入f函数，f返回false则跳出循环，否则继续遍历map

Delete方法

删除map中的键值

sync.Map操作流程图

适用场景

从代码中看出，初始化dirty时，会将read中所有未删除的数据复制到dirty，开销很大，而频繁往map中插入新数据会导致dirty中有大量read中没有的数据，从而导致read的命中率过低，需要频繁调用锁进行操作，并且未命中次数达到len(dirty)后，dirty会被升级为read，再次有新数据插入的时候，又会重复dirty初始化的过程，开销很大。但若读的操作很多，写的很少，大量的操作，无需用锁，用原子操作就可以完成，比较高效，也会降低dirty初始化的频率。

所以sync.Map适用于读多写少的场景。
